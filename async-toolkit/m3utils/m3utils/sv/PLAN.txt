SystemVerilog Parser Frontend (svfe) -- Project Plan
=====================================================

Mika Nystroem, March 2026


1.  MOTIVATION AND GOALS

    We want a standalone SystemVerilog parser frontend, built in
    Modula-3 using the CM3 parserlib metacompiler, with Scheme
    (mscheme) as the extension/scripting language.  The parser
    should understand enough SystemVerilog to support real ASIC
    design automation tasks:

      - Linting (undriven signals, width mismatches, latch inference)
      - Elaboration (parameter resolution, hierarchy flattening)
      - Gate-level decomposition (silicon compilation)
      - Custom design rule checking and transformations via Scheme

    The target language level is the SystemVerilog synthesis subset
    as supported by Synopsys Design Compiler and Cadence Genus --
    roughly IEEE 1800-2017 minus the verification/OOP features.
    Object-oriented constructs (classes, constraints, covergroups)
    are explicitly excluded since they do not help ASIC synthesis.


2.  ARCHITECTURE

    The architecture follows the CSP compiler (csp/) exactly:

      sv/
      +-- svgrammar/src/       AST type definitions (M3 interfaces)
      +-- svparse/src/         Parser generator input files
      |     sv.t               Token definitions
      |     sv.l               Lexer specification (DFA)
      |     sv.y               YACC grammar (LR(1))
      |     svLexExt.e         Lexer extension (captures token text)
      |     svParseExt.e       Parser extension (semantic actions)
      |     Main.m3            Command-line driver (svfe program)
      +-- src/                 Scheme-integrated compiler (svpc, later)
      +-- doc/                 Documentation

    How it works (same pipeline as cspfe):

      Source file (.sv/.v)
        --> Preprocessor (handles `define, `include, `ifdef)
        --> Lexer (sv.l --> DFA, tokenizes into sv.t tokens)
        --> Parser (sv.y --> LR(1), builds parse tree)
        --> Extension (svParseExt.e, semantic actions)
        --> S-expression output (consumed by Scheme)
        --> Scheme scripts do elaboration, linting, synthesis

    The parserlib toolchain generates the lexer and parser code
    from declarative specifications:

      ktok  processes sv.t  -->  svTok.i3    (token type interface)
      klex  processes sv.l  -->  svLex.i3/m3 (DFA lexer)
      kyacc processes sv.y  -->  svParse.i3/m3 (LR(1) parser)
      kext  processes .e    -->  svLexExt, svParseExt (extensions)

    The m3makefile for svparse/src/ will contain:

      import ("libm3")
      import ("cit_util")
      import ("parserlib")

      Token("sv")
      Lexer("sv","sv")
      Parser("sv","sv")

      Extended("svLexExt")
      Extended("svParseExt")

      implementation ("Main")
      program ("svfe")


APPENDIX A: DESIGN-ANNOTATION SUPPORT
======================================

Based on: /Users/mika/sky2-git/docs/svfe_wishlist.md

Three features: (1) source line numbers in AST, (2) annotation
comment tokens, (3) translate_off/on handling.


Architecture Context
--------------------

The svfe lexer is generated by klex from sv.l, producing a DFA.  The
parser is generated by kyacc from sv.y, producing an LALR automaton.
Extensions (.e files processed by kext) inject Modula-3 semantic
actions.

Key constraint: the skip rule in sv.l is a single DFA pattern
matching whitespace + block comments + line comments.  It consumes
ALL comments including annotations.  We cannot simply add a
higher-priority T_ANNOTATION rule because skip's Kleene star
produces a longer match.

Key constraint: $N in semantic actions gives the .val field (TEXT)
of the Nth value-carrying child.  ConstTokens (keywords) are skipped
in $N numbering.  There is no built-in way to access a child's line
number through $N.


A.1  LEXER LINE TRACKING
-------------------------

Goal: every token returned by the lexer carries an accurate source
line number, accessible via self.curLine on the lexer object.

Approach: override the get() method in svLexExt.e.  After calling
the parent svLex.T.get(self), count newlines in the skip region
(between the end of the previous token and the start of the current
token) by seeking through the Rd.

Files changed:
  svparse/src/svLexExt.e

Changes:

  1. Add %public section:

       %public {
         curLine : INTEGER := 1;
         prevEnd : INTEGER := 0;
       }

  2. Add %overrides section:

       %overrides {
         get := MyGet;
       }

  3. Add MyGet procedure in %module:

       PROCEDURE MyGet(self: T): Token RAISES {Rd.EndOfFile} =
         VAR
           tok    : Token;
           rd     := self.getRd();
           curPos : INTEGER;
           tokLen : INTEGER;
           skipEnd: INTEGER;
         BEGIN
           tok := svLex.T.get(self);
           curPos := Rd.Index(rd);
           tokLen := Text.Length(self.getText());
           skipEnd := curPos - tokLen;
           (* Count newlines in the skip region *)
           IF skipEnd > self.prevEnd THEN
             Rd.Seek(rd, self.prevEnd);
             FOR i := self.prevEnd TO skipEnd - 1 DO
               IF Rd.GetChar(rd) = '\n' THEN
                 INC(self.curLine)
               END
             END
           END;
           Rd.Seek(rd, curPos);
           self.prevEnd := curPos;
           RETURN tok
         END MyGet;

     This counts newlines in whitespace+comments between tokens,
     giving curLine = line of the first character of the returned
     token.

     Works because Main.m3 uses FileRd.Open (not SeekRdClass),
     so SeekRd.DiscardPrevious is a no-op and backward seeks are
     safe.


A.2  SPLIT COMMENT HANDLING + NEW TOKEN TYPES
----------------------------------------------

Goal: make annotation comments and translate_off/on directives
visible to the parser as distinct tokens, while still skipping
regular comments.

Approach: split the monolithic skip rule so that line comments
become explicit tokens whose extension handlers decide
skip-or-emit.

Files changed:
  svparse/src/sv.l
  svparse/src/sv.t
  svparse/src/svLexExt.e

--- sv.l changes ---

Replace:
  skip  ([ \t\r\n]|("/*"(...)*"*"+"/")|("//"[^\n]*"\n"))*

With:
  T_LINE_COMMENT   "//"[^\n]*"\n"
  T_BLOCK_COMMENT  "/*"([^*]|"*"+[^*/])*"*"+"/"
  skip             [ \t\r\n]*

(skip now matches only whitespace; comments are explicit tokens.)

Add dummy patterns for virtual tokens (never matched by DFA,
but need codes allocated by klex for the parser to reference):
  T_ANNOTATION     "__ANNOTATION__"
  T_TRANSLATE_OFF  "__TRANSLATE_OFF__"
  T_TRANSLATE_ON   "__TRANSLATE_ON__"

--- sv.t changes ---

Add to %token line:
  T_LINE_COMMENT T_BLOCK_COMMENT T_ANNOTATION
  T_TRANSLATE_OFF T_TRANSLATE_ON

--- svLexExt.e changes ---

Add token type declarations and handlers:

  T_LINE_COMMENT: {}
  T_LINE_COMMENT {
    VAR
      text := $;
      len  := Text.Length(text);
      i    : INTEGER;
      kw   : TEXT;
    BEGIN
      i := 2;
      WHILE i < len AND (Text.GetChar(text, i) = ' '
                         OR Text.GetChar(text, i) = '\t') DO
        INC(i)
      END;
      IF i < len AND Text.GetChar(text, i) = '@' THEN
        RETURN NEW(T_ANNOTATION, val := text)
      END;
      kw := "synopsys";
      IF i + Text.Length(kw) <= len
         AND Text.Equal(
               Text.Sub(text, i, Text.Length(kw)), kw) THEN
        INC(i, Text.Length(kw));
        WHILE i < len AND (Text.GetChar(text, i) = ' '
                           OR Text.GetChar(text, i) = '\t') DO
          INC(i)
        END;
        IF StartsWith(Text.Sub(text, i), "translate_off") THEN
          RETURN NEW(T_TRANSLATE_OFF, val := text)
        ELSIF StartsWith(Text.Sub(text, i), "translate_on") THEN
          RETURN NEW(T_TRANSLATE_ON, val := text)
        END
      END;
      RETURN NIL   (* regular comment -- skip *)
    END
  }

  T_BLOCK_COMMENT: {}
  T_BLOCK_COMMENT { RETURN NIL }

  T_ANNOTATION: { val : TEXT }
  T_TRANSLATE_OFF: { val : TEXT }
  T_TRANSLATE_ON: { val : TEXT }

When handlers return NIL, the generated Get() loop continues
scanning (same as skip).  When they return a real token, it
reaches the parser.

Line counting note: MyGet wraps the entire svLex.T.get(self)
call, which loops internally over skip and NIL-returning tokens.
All newlines in skipped comments are counted in the skip region
between prevEnd and the start of the finally-returned token.


A.3  LINE NUMBERS IN THE AST
------------------------------

Goal: key AST nodes (module_item, description level) carry source
line numbers as (@ LINE ...) wrappers.

Approach: add a `mark` epsilon production that captures curLine at
the point where the parser commits to a module_item or description.

In LALR parsing, an epsilon production at the start of every
alternative reduces BEFORE the first real token is shifted.  At
that point, the lookahead token has already been fetched, so
curLine reflects that token's line -- exactly the first token of
the construct.

Files changed:
  svparse/src/sv.y
  svparse/src/svParseExt.e
  svparse/src/Main.m3

--- svParseExt.e changes ---

  1. Add lexer reference in %public:

       %public {
         scmResult : TEXT;
         lexer     : svLexExt.T;
       }

  2. Add imports in %module:

       IMPORT Fmt, svLexExt;

  3. Add mark production action:

       mark: { val : TEXT; cnt : INTEGER; }
         empty  { $$.val := Fmt.Int(self.lexer.curLine) }

  4. Add WrapLine helper in %module:

       PROCEDURE WrapLine(line, body : TEXT) : TEXT =
         BEGIN
           RETURN "(@ " & line & " " & body & ")"
         END WrapLine;

  5. Update module_item actions.  Every alternative gets mark as
     first symbol; $N indices shift by +1.

     BEFORE:
       port_dir_decl   { $$.val := $1 }
       always_block    { $$.val := $1 }
       ...

     AFTER:
       port_dir_decl   { $$.val := WrapLine($1, $2) }
       always_block    { $$.val := WrapLine($1, $2) }
       ...

  6. Similarly update description actions.

--- sv.y changes ---

  1. Add mark production:

       mark:
         empty

  2. Insert mark before every alternative in module_item:

     BEFORE:
       module_item:
         port_dir_decl    port_direction_declaration ';'
         net_decl         net_declaration ';'
         always_block     always_construct
         ...

     AFTER:
       module_item:
         port_dir_decl    mark port_direction_declaration ';'
         net_decl         mark net_declaration ';'
         always_block     mark always_construct
         ...

  3. Insert mark before every alternative in description.

--- Main.m3 changes ---

  Set the lexer reference on the parser before parsing:

    parser.lexer := lexer;


A.4  ANNOTATION GRAMMAR PRODUCTIONS
-------------------------------------

Goal: annotation comments (// @keyword ...) wrap the next
module_item in the AST.

Files changed:
  svparse/src/sv.y
  svparse/src/svParseExt.e

--- sv.y ---

Add to module_item alternatives:

  annotation       T_ANNOTATION module_item

Multiple annotations stack via recursion: a second T_ANNOTATION
matches inside the recursive module_item.

Note: no mark before T_ANNOTATION -- T_ANNOTATION is a real
token (not epsilon), and the line is already in the annotation
text.

--- svParseExt.e ---

Add to module_item:

  annotation  { $$.val := "(annotation " & $1 & " " & $2 & ")" }

$1 = T_ANNOTATION.val (full comment text including "// @...").
$2 = module_item.val (the annotated construct, with its own
     (@ LINE ...) wrapper from mark).


A.5  TRANSLATE_OFF/ON GRAMMAR PRODUCTIONS
------------------------------------------

Goal: items between // synopsys translate_off and translate_on
are grouped in a (translate_off ...) AST node.

Files changed:
  svparse/src/sv.y
  svparse/src/svParseExt.e

--- sv.y ---

Add to module_item alternatives:

  translate_off  mark T_TRANSLATE_OFF translate_off_body T_TRANSLATE_ON

Add new production:

  translate_off_body:
    empty
    cons           translate_off_body module_item

--- svParseExt.e ---

  translate_off_body: { val : TEXT; cnt : INTEGER; }
    empty  { $$.val := "" }
    cons   { $$.val := Seq($1, $2) }

  In module_item:
    translate_off
      { $$.val := WrapLine($1, "(translate_off " & $2 & ")") }

  $1 = mark.val (line number).
  $2 = translate_off_body.val.
  T_TRANSLATE_OFF and T_TRANSLATE_ON are skipped in $N numbering
  since they are value-carrying %tokens (not ConstTokens) -- hmm,
  actually they ARE value-carrying, so they DO count.  Need to
  verify $N mapping at implementation time.  If they count:
  $1=mark, $2=T_TRANSLATE_OFF.val, $3=translate_off_body.val.


A.6  `line DIRECTIVE SUPPORT
-----------------------------

Goal: correct line numbers when macro expansion changes line
counts.

Problem: svpp preserves line counts for directive lines (emits
blanks) and for multi-line macro INVOCATIONS (pads output to
match source line count).  But when a single-line macro
invocation expands to MULTIPLE output lines, svpp emits all of
them, shifting every subsequent line.  Example:

  Source line 10:  `EXPAND_TO_FIVE_LINES(x)
  Output lines 10-14: five lines of expanded code
  Source line 11 now appears at output line 15.

So `line directives are needed NOW, not just for future include
inlining.

--- svpp changes ---

After expanding macros (line 811 in Main.m3), when outLines >
srcLines, emit a `line directive to resynchronize:

  (* After the output loop at line 825 *)
  IF outLines > srcLines THEN
    (* Line count shifted -- emit resync directive.
       Next source line number = current source line + srcLines *)
    EmitLine("`line " & Fmt.Int(nextSourceLine)
             & " \"" & filename & "\" 0")
  END;

This requires tracking the current source line number through
the Preprocess loop (add a lineNo counter incremented for each
source line consumed).

Also emit `line at `include boundaries (even though includes
currently don't inline content, this prepares for that).

--- svfe (svLexExt.e) ---

The T_DIRECTIVE token already matches `line directives.  In
MyGet, after getting a T_DIRECTIVE token, check if it starts
with `line.  If so, parse line number and filename, update
self.curLine and self.curFile, and recurse to get the real next
token.

  Add to %public:
    curFile : TEXT := "";

  In MyGet, after tok := svLex.T.get(self):
    TYPECASE tok OF
    | T_DIRECTIVE(d) =>
      IF IsLineDirective(d.val) THEN
        ParseLineDirective(self, d.val);
        RETURN MyGet(self)
      END
    ELSE END;

--- svpp ---

svpp preserves line counts for directives and multi-line macro
invocations, but NOT when a macro expands to more lines than
the invocation.  See A.6 for the fix: emit `line directives
after such expansions to resynchronize.
